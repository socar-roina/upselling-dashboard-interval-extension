"당일 출발" 조건의 영향력 (5/14 실측)
5/14 왕복 예약 전체12,900건
↓ "예약한 날 바로 차를 빌리는" 경우만 남기면
당일 출발 예약8,107건 (전체의 62.8%)
이 한 조건만으로 4,793건 걸러짐. 이후 시간대 조건까지 적용하면 573건까지 좁아짐. 즉 절대 수치가 작은 건 노출 정책의 의도된 결과.
4
시간/요일 조건 부합 모수는 530명 — 절대 수치 적은 게 노출 정책 결과로 보임
빅쿼리에서 시간/요일/요금제 조건까지 적용하면 5/14에 530명까지 좁힐 수 있음. 이후 차종 제외 / 차량 가용 / 추가금 조건이 추가로 적용되면 실제 36명까지 자연스럽게 좁아질 수 있는 구조. 정확한 단계별 통과 수치 추적은 추후 노출 시점 로그 보강으로 가능.
5
결제 후 환불된 케이스 분리 필요
한 예약에서 결제 후 환불되면 결제 내역에 결제(+)와 환불(-) 두 행이 모두 들어옴. 한 예약 단위로 합계가 양수일 때만 진짜 수락으로 봐야 정확. 분석 기간 동안 결제 16건 중 7건이 환불되어 실제 수락 9건. 자세한 내용은 위 "결제와 환불 분해" 섹션 참고.
필요한 조치 (우선순위순)
위 5가지 사실 정리에서 도출된 조치 항목. 즉시 처리할 것, 이번 주 안에 BE 확인할 것, 1주 후 재평가할 것 으로 구분.
1
수락 카운트는 환불 분리 후 산정
즉시
결제 내역에 결제(+)와 환불(-) 행이 같이 들어오므로, 한 예약 단위로 결제 합계가 양수일 때만 수락으로 카운트. 분석 기간 누계 결제 16건 중 환불 7건 발생 → 실제 수락 9건.
2
환불(취소)율 가드레일 추가
즉시
시간연장 결제 후 환불되는 비율을 일별로 추적. 너무 높으면 정책/UX 점검 필요. 환불 사유는 운영팀에 별도 확인.
현재 추세 (실측)
5/13 환불율100% (3건 결제 → 3건 다 환불)
5/14 환불율30.8% (13건 결제 → 4건 환불)
계산식환불된 예약 수 ÷ 결제된 예약 수
표본 작아 변동성 큼 — 1주 누적 후 정확한 추세 판단 필요
3
BE팀에 노출 모수 정합성 확인 요청
이번 주 안
빅쿼리로 확인되는 530명 → 실제 옵션 화면 본 36명 사이 단계별 통과율을 BE에서 측정해서 공유 요청.
BE팀에 보낼 요청 사항
요청 15/14 시간/요일 조건 부합 530명 중 실험군(50%)으로 분류된 회원 수
요청 2차종 제외(수입차/전기차/대형) 후 잔여 회원 수
요청 3차량 가용 (자정까지 비어있음) 통과 회원 수
요청 4추가 요금 4,900원 초과 조건 통과 회원 수
요청 5제휴 할인 / 쿠폰 호환 통과 회원 수
4
실험군/대조군 나누는 기준 일치 여부 검증
이번 주 안
빅쿼리에서 회원을 실험군으로 분류할 때 쓰는 기준과 노출 시스템에서 쓰는 기준이 일치해야 A/B 비교가 의미가 있음. 두 기준이 다르면 BQ 기반 A/B 비교가 무의미해짐 → BE 확인 필요.

사용된 빅쿼리 쿼리 (직접 검증용)
위 표/차트에 들어간 숫자를 직접 검증할 때 사용 가능한 빅쿼리 쿼리. 그대로 실행하면 같은 결과 도출.
1. 화면별·동작별 일일 집계 쿼리
"옵션 화면 본 회원", "옵션 카드 누른 회원", "결제 버튼 누른 회원" 등을 일별로 집계
SELECT DATE(event_at_kst) AS dt, COUNT(DISTINCT CASE WHEN event_name='view' AND component_name IS NULL THEN member_id END) AS upsell_view, COUNT(DISTINCT CASE WHEN event_name='click' AND component_name='interval_option' THEN member_id END) AS interval_click, COUNT(DISTINCT CASE WHEN event_name='click' AND component_name='cta_button' THEN member_id END) AS cta_click FROM `socar-data.app_web_log.socar_app_web_log` WHERE event_date_kst BETWEEN "2026-05-13" AND "2026-05-14" AND event_at_kst >= DATETIME("2026-05-13 14:09:30") AND page_name = 'upsell_bottomsheet' GROUP BY dt ORDER BY dt
2. 정확한 수락 건수 쿼리 (환불 제외)
한 예약 단위로 결제 합계가 양수인 경우만 진짜 수락으로 카운트
WITH net_data AS ( SELECT DATE(created_at, "Asia/Seoul") AS dt, reservation_id, SUM(amount) AS net_amount FROM `socar-data.dbt.stg_tianjin_replica__paid_info` WHERE DATE(created_at, "Asia/Seoul") BETWEEN "2026-05-13" AND "2026-05-14" AND paid_type = 'upselling_discount' AND DATETIME(created_at, "Asia/Seoul") >= DATETIME("2026-05-13 14:09:30") GROUP BY dt, reservation_id ) SELECT dt, SUM(CASE WHEN net_amount > 0 THEN 1 ELSE 0 END) AS net_accepted_rsv, SUM(CASE WHEN net_amount > 0 THEN net_amount ELSE 0 END) AS company_burden_net, SUM(CASE WHEN net_amount > 0 THEN 4900 ELSE 0 END) AS revenue_increment FROM net_data GROUP BY dt ORDER BY dt

대시보드 용어 정리

옵션 화면 (업셀링 바텀시트)결제 직전에 화면 아래에서 올라오는 시간연장 옵션 카드가 뜬 화면. 사용자가 이 화면을 봐야만 시간연장이 가능.
시간연장 카드 누르기 (옵션 토글)옵션 화면에서 사용자가 "연장 시간으로 변경" 카드를 직접 눌러서 선택한 행위. 결제 의도 표시.
결제 버튼 누르기옵션 화면 맨 아래 결제 진행 버튼 누른 행위. 시간연장을 선택했든 안 했든 결제로 넘어가는 의미라 "수락" 과는 다름.
실제 수락한 예약의 결제 합계가 양수인 경우. 결제 후 환불되지 않고 끝까지 유지된 경우만 카운트.
매출 증분사용자가 시간연장 옵션 수락으로 추가로 낸 돈. 한 건당 4,900원 고정 × 수락 건수.
회사 할인 부담실제 연장 요금 - 사용자가 낸 4,900원. 차액은 회사가 할인 형태로 부담함. 수락 건들의 차액 합계.
수락률옵션 화면을 본 회원 중 실제로 수락한 회원의 비율. 실제 수락 회원 수 ÷ 옵션 화면 본 회원 수.
실험군 / 대조군 (A/B)실험군(B군) = 옵션 화면을 보는 회원, 대조군(A군) = 옵션 화면이 안 뜨는 회원. 50:50으로 무작위 분배.
